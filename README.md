### Архитектурная «коробка»

1. **Единый сервер на FastAPI**
    
    - **Бот (aiogram):** Отвечает только за интерфейс, прием сообщений от пользователя и первичную валидацию команд.
    
    - **Бэкенд (FastAPI):** Хранит базу данных, обрабатывает тяжелые вычисления, управляет бизнес-логикой.
    
    - **Общение:** Бот делает запросы к FastAPI, получает данные и показывает их пользователю.
        
2. **База данных — SQLite**
    
    - Файл рядом с проектом, доступ через «простую» библиотеку (вероятно, синхронный `sqlite3).
        
3. **Админ и подписчик**

    - Админ работает через **веб-оболочку** (рендеринг форм от FastAPI) — заполняет темы, статьи, вопросы, сохраняет.
        
    - Подписчик взаимодействует **только через Telegram-бота**: получает статьи, проходит тесты с вариантами ответов.
        

### Поток данных (упрощённо)

- Админ заходит на веб-интерфейс, заполняет формы, нажимает «сохранить».
    
- FastAPI обрабатывает данные, пишет в SQLite.
    
- Когда приходит время рассылки (по кнопке админа или расписанию), бот отправляет статью подписчикам через вызовы Telegram API.
    
- Подписчик отвечает на тест → FastAPI/aiogram получает апдейт → сверяет ответ с базой → отвечает результатом.


Часть 1. База данных и слой доступа (асинхронный)
Цель: создать работающее подключение к SQLite, реализовать CRUD для всех таблиц.

Что сделаем:

Файл config.py — настройки (путь к БД, токен бота, секрет админки)

Файл database.py — пул соединений sqlite3, базовые функции execute, fetch_all, fetch_one

Файл models.py — Pydantic-схемы для всех сущностей (Category, Article, Question, Answer, User, History, MailingLog)

Файл crud.py — асинхронные функции для каждой таблицы:

create_category, get_categories, update_category, delete_category

create_article, get_articles, get_article_by_id, update_article, delete_article

create_question, get_questions_by_article, create_answer, get_answers_by_question

register_user, get_user_by_tg_id, set_user_active

save_user_answer, get_user_history_for_article

log_mailing, get_unsent_users_for_article

Проверка: напишем простой скрипт test_db.py, который создаст пару категорий, статей, вопросов и убедится, что всё пишется/читается.

Часть 2. FastAPI – базовые эндпоинты (без админки)
Цель: дать возможность боту и вебу получать данные через API.

Структура:

main.py — FastAPI приложение, роутеры

routers/articles.py — GET /articles, GET /articles/{id}

routers/users.py — POST /users/register, GET /users/me/history

routers/quiz.py — POST /quiz/check (принимает {user_id, question_id, answer_id})

routers/mailing.py — POST /mailing/send (пока заглушка для админа)

Детали:

Все эндпоинты возвращают JSON.

Используют crud.py для работы с БД.

В /quiz/check вычисляем is_right и сохраняем в UserHistory.

Добавим простой API‑ключ (X-API-Key) для защиты эндпоинтов, которые будут вызывать бот и админка.

Проверка: через Swagger (/docs) проверить создание пользователя, получение статьи с вопросами, проверку ответа.

Часть 3. Telegram‑бот (aiogram 3.x)
Цель: пользователь получает статьи и проходит тесты.

Что делаем:

bot_main.py — запуск бота, диспетчер, регистрация хендлеров

handlers/start.py — команда /start → вызывает POST /users/register бэкенда

handlers/menu.py — кнопка "Статьи" → GET /articles → показывает список заголовков

handlers/article.py — выбор статьи → GET /articles/{id} → показывает текст и кнопку "Пройти тест"

handlers/quiz.py — по очереди показывает вопросы с кнопками-ответами → после каждого ответа отправляет POST /quiz/check → если правильно, идёт дальше; в конце выводит результат

Важно:

Хранить состояние теста (какой вопрос сейчас) в FSMContext aiogram.

Никаких прямых запросов к БД, только через API бэкенда.

Проверка: запустить бота и пройти полный цикл от /start до последнего вопроса.

Часть 5. Рассылки (ручная + авторасписание)
Цель: бот рассылает статьи подписчикам, логирует отправку.

Что делаем:

Дорабатываем routers/mailing.py: при POST /mailing/send

получаем article_id

из БД берём всех активных пользователей (is_active=1)

для каждого проверяем MailingLog, не отправлялась ли уже эта статья этому пользователю

отправляем задание боту: как? – варианты:

Бот имеет отдельный HTTP‑эндпоинт (например, /bot/send_article), который вызывает API Telegram. Тогда бэкенд вызывает его для каждого пользователя. Проще всего.
Бот периодически опрашивает бэкенд на предмет новых рассылок – сложнее.
Логируем отправку в MailingLog

Фоновый планировщик (например, apscheduler) раз в минуту проверяет статьи с schedule_at <= now и статусом published и запускает для них ту же процедуру рассылки.

Проверка: создать статью с расписанием на будущее, дождаться, бот отправит всем пользователям.
